# Baton

Claude Code를 단계별로 실행하고 검증하는 Supervisor Harness.

단계 정의(steps)를 입력하면 Claude Code가 각 단계를 순차적으로 수행하고, Supervisor가 결과를 검증한 뒤 다음 단계로 진행한다. 진행 상태는 외부 상태머신으로 관리되며, 모든 단계의 성공 여부는 실제 검증 결과를 기준으로 판단한다.

## 왜 만들었나

Claude Code는 단일 작업 수행에는 강력하지만, 여러 단계에 걸친 작업을 장시간 안정적으로 수행하려면 진행 상태 관리, 검증, 실패 제어를 담당하는 별도의 계층이 필요하다.

Baton은 이러한 문제를 해결하기 위해 만든 Supervisor Harness로, Claude Code를 반복 호출하면서 상태를 관리하고 검증 결과를 기준으로 다음 단계를 결정한다.

> **참고**
>
> 자율 실행 루프 아이디어는 오픈소스 프로젝트 `sidabari4loop`에서 영감을 받았다.
>
> 다만 sidabari4loop가 PTY 세션과 Hook 이벤트 기반으로 동작하는 반면, Baton은 `claude -p` 기반의 헤드리스 단발 호출 방식을 사용한다.

## 동작 흐름

```text
Load state
    ↓
Run Claude
    ↓
Verify
    ↓
Update state
    ↓
Guard check
    ↓
Next step / Halt
```

각 루프는 다음 순서로 진행된다.

1. `state.json`에서 현재 단계 로드
2. `claude -p`로 해당 단계 실행
3. Supervisor가 검증 명령(`npm test` 등) 직접 수행
4. 성공 시 다음 단계 진행
5. 실패 시 실패 횟수 누적
6. Guard가 중단 여부 판정

## Claude Code 네이티브 기능과의 관계

Claude Code는 이미 에이전트 루프, 비용 정보 출력, 예산 및 턴 제한 기능을 제공한다.

Baton은 이러한 기능을 대체하기보다 Supervisor 관점의 검증과 상태 관리를 추가하는 데 초점을 둔다.

### 1. Supervisor 독립 검증

네이티브 Claude Code도 지시하면 스스로 테스트를 실행한다. 차이는 테스트를 돌리느냐가 아니라 **검증 주체의 위치**다.

네이티브는 작업하는 주체와 합격을 판정하는 주체가 같은 에이전트 루프 안에 있다 — 자기가 구현하고 자기가 통과를 판정한다. Baton은 판정 주체를 에이전트 바깥의 Supervisor로 분리해, 단계 종료 후 Supervisor가 독립적으로 검증 명령을 실행하고 그 결과로 합격을 판정한다. 에이전트가 완료를 보고해도 외부 검증이 실패하면 다음 단계로 진행하지 않는다.

### 2. 외부 상태머신

네이티브는 `--resume` / `-c`로 이전 세션을 이어갈 수 있으나, 이는 누적된 대화 트랜스크립트(컨텍스트)를 복원하는 방식이다. 장시간 실행일수록 컨텍스트가 계속 불어난다.

Baton은 매 턴 새로운 컨텍스트로 시작하고, 진행 상태만 구조화해 `state.json`에 외부화한다. 즉 컨텍스트를 누적하지 않고도 작업을 이어간다. 이는 아래 "한계 및 개선 방향"의 컨텍스트 누적 문제를 의도적으로 회피하기 위한 설계 선택이다.

### 3. 의미 기반 중단(HALT)

리소스 한도 외에도 연속된 검증 실패를 중단 조건으로 사용한다.

예산이나 턴 수가 남아 있더라도 동일 단계에서 반복적으로 실패하면 자동으로 실행을 중단한다.

## 주요 기능

### Headless Execution

`claude -p`를 사용해 매 턴 독립적으로 실행한다.

PTY 세션을 유지하지 않으며, 진행 상태는 외부 상태머신을 통해 관리한다.

### Step Verification

각 단계 종료 후 Supervisor가 직접 검증을 수행한다.

검증 실패 시 다음 단계로 진행하지 않는다.

### Cost Tracking

토큰 사용량과 비용을 모델별로 집계한다.

작업 종료 후 단계별 실행 시간, 토큰 사용량, 비용 정보를 포함한 리포트를 생성한다.

### HALT Guard

연속 실패, 예산 초과, 최대 턴 초과를 감지해 자동으로 실행을 중단한다.

## 검증 결과

### 정상 완료 시나리오

URL 단축 API를 단계 정의만으로 생성하는 End-to-End 시나리오를 수행했다.

- 8개 단계 모두 성공
- 최종 상태: `complete`
- 총 실행 시간 약 6분 32초
- API 환산 비용 약 $1.90

#### 결과물

- Repository: `<repository-url>`
- Report: `<report-url>`

### 실패 시나리오 검증

의도적으로 실패하도록 작성한 테스트를 포함한 단계 정의를 사용해 Supervisor 검증 및 HALT 가드 동작을 검증했다.

관찰 결과:

- 에이전트가 완료를 보고하더라도 검증 실패 시 성공으로 처리되지 않음
- 동일 단계에서 3회 연속 실패 시 자동 중단
- 이후 단계로 진행되지 않음
- 중단 사유가 상태 파일에 기록됨

#### 증거

```text
examples/halt-demo/
```

## 구조

| Module | Responsibility |
|----------|----------|
| `claude.ts` | Claude CLI 호출 래퍼 |
| `state.ts` | 상태머신 및 진행 상태 관리 |
| `verify.ts` | 검증 실행 및 자동 커밋 |
| `guard.ts` | HALT 판정 |
| `telemetry.ts` | 비용 집계 및 리포트 생성 |
| `supervisor.ts` | 실행 루프 오케스트레이션 |

## 실행

```bash
npm install

npx tsx index.ts <target-folder> <steps-file>

# Example
npx tsx index.ts ../url-shortener ./url-shortener-steps.js
```

## 테스트

```bash
npm test
```

핵심 제어 로직에 대해 총 20개의 단위 테스트를 작성했다.

검증 범위는 다음과 같다.

- 상태머신 단계 전이
- HALT 가드 조건 및 우선순위
- 모델별 비용 집계

상태 파일의 저장·복구와 실행 계층(Claude 호출, Supervisor 루프)은 정상 완료 및 실패 시나리오의 End-to-End 실행을 통해 검증했다.

## 한계 및 개선 방향

현재는 수 분 규모의 프로젝트 생성 작업을 중심으로 검증했다.

장시간 실행 환경에서는 다음과 같은 보완이 필요하다.

- 컨텍스트 관리 전략 고도화
- 단계 실패 복구 메커니즘
- 재시도 정책 개선
- 장기 실행 환경에서의 안정성 검증

향후 이러한 부분을 중심으로 개선할 계획이다.
